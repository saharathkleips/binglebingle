# Milestone: Game Layout

**Status:** Pending
**Affected Slices:** `src/components/app/`, `src/components/history-area/`, `src/components/submission-area/`, `src/components/pool/`

## Context

빙글빙글 currently uses a simple centered flex column: NavBar at top, then HistoryArea, SubmissionArea, and Pool stacked vertically with 1.5rem gap and 2rem padding. This layout doesn't utilize screen real estate well on mobile and doesn't support the "playable canvas" concept where tiles can be freely arranged.

**Prerequisite:** 1.4.2 (Tile Visuals) must be complete — tile dimensions are finalized so layout can be built around real card sizes.

**Current layout** (`App.tsx`):

```
NavBar
div (flex-column, centered, gap 1.5rem, padding 2rem)
  HistoryArea
  GameContent (SubmissionArea + Pool, or WinPanel if won)
```

**Stack:** CSS Modules + native CSS. React 19. Mobile-first, responsive for larger screens. Most players will be on phones.

## Goal

Replace the stacked flex-column layout with a mobile-first three-region layout optimized for gameplay: History at the top (grows upward as guesses accumulate), SubmissionArea in the middle, and Pool (playable canvas) at the bottom filling remaining space.

## Success Criteria

1. On a 375x667 viewport (iPhone SE): History at top, submission in the middle, pool canvas fills the bottom.
2. No outer page scrolling — the game fills the viewport exactly (`100dvh`).
3. Pool area fills all remaining vertical space after history and submission.
4. History region shows 2-3 most recent guesses; older guesses are accessible by scrolling that specific region (not the whole page).
5. All existing drag interactions (tap, drag-to-slot, drag-to-tile) still work with the current pointer-event system.
6. Layout is responsive — works on both small phones and larger screens.

## Design Notes

**Layout rationale — upward flow:** Tiles live in the pool canvas at the bottom. The player drags tiles up into submission slots in the middle. When a guess is submitted, the result flows upward into the history region at the top — this will be animated in 1.4.5. This creates a natural upward momentum: canvas → submission → history.

**Visual stacking order (top to bottom of screen):**

```
┌─────────────────────┐
│ Guess history       │  ← 2-3 rows visible, scrollable
│ (newest at bottom,  │     for older guesses
│  grows upward)      │
├─────────────────────┤
│ Submission slots    │  ← always visible, fixed middle
│ [Submit button]     │
├─────────────────────┤
│                     │
│   Pool / Canvas     │  ← fills remaining space
│   (tile play area)  │
│                     │
└─────────────────────┘
```

**`100dvh`** (dynamic viewport height) accounts for mobile browser chrome (address bar, bottom nav). This avoids the classic "100vh is too tall on mobile" issue.

**Future considerations:**

- 1.4.4 will use the pool container as the `bounds` for GSAP Draggable — make sure the pool has a stable, measurable DOM element.
- Games with many tiles may need the canvas to extend beyond the viewport. This milestone establishes the bounded canvas container; expandability can be added later.
- The submit-then-animate-upward flow will be implemented in 1.4.5. For now, guesses just appear in the history region after submission.
