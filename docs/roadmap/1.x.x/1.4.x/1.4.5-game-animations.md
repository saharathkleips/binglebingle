# Milestone: Game Animations

**Status:** Complete
**Affected Slices:** `src/lib/animation/`, `src/components/pool/`, `src/components/submission-area/`, `src/components/history-area/`

## Context

빙글빙글 is a Korean word-guessing game with a hwatu (화투) card aesthetic established in 1.4.1–1.4.2. Since we're building a game (not a marketing page), animations should spark joy and feel inspired — tactile, physical, and decisive, like playing cards on a table.

**Prerequisites:**

- 1.4.1 (Design Tokens) — palette for visual effects
- 1.4.2 (Tile Visuals) — final tile appearance
- 1.4.3 (Game Layout) — spatial layout (pool canvas, submission at bottom, history above submission)
- 1.4.4 (GSAP Drag) — GSAP installed, Draggable wired, animation utility layer exists with `animatePickUp`, `animatePutDown`, `animateReposition`

**Game actions that need animation:**

- Picking up a tile (drag start) — done in 1.4.4
- Putting down a tile (drag end, valid drop) — done in 1.4.4
- Repositioning a tile (drag end, invalid drop / rejected compose) — done in 1.4.4
- Composing two tiles (drag tile onto tile, successful merge)
- Decomposing a tile (tap to split into components)
- Rotating a tile (tap to cycle jamo)
- Submitting a tile to a submission slot (drag from pool to slot)
- Swapping slot tiles (drag filled slot to filled slot) — basic animation in 1.4.4, polish here
- Submitting a guess (pressing the submit button)
- Revealing guess correctness (character-by-character result reveal)

**Stack:** GSAP (installed in 1.4.4) + CSS Modules + React 19.

## Goal

Add micro-interaction animations for all remaining game actions, plus explore particle effects, hover states, and glow treatments to elevate the game feel.

## Success Criteria

1. Every game action has a visible, polished animation that feels intentional.
2. Animations feel inspired by hwatu card play — physical, tactile, snappy, decisive.
3. Animations do not block gameplay — state dispatch remains synchronous, animations are reactive.
4. Guess reveal flows naturally: submitted tiles animate upward into history, flip to show results, older guesses scroll up.
5. All existing tests pass. E2E tests account for animation timing before asserting final state.
6. Animation durations and easings are tunable through the animation utility layer.
7. At least one particle or glow effect is implemented and feels natural (VIS-28/29 are exploratory — partial delivery is fine).

## Design Notes

**State coordination:** Dispatch remains synchronous (the current model). Animations are triggered reactively after state changes:

- `useLayoutEffect` detects state changes (new/changed/removed tiles) and plays entrance/exit animations.
- Reveal animation is keyed off `history.length` change in HistoryArea.
- This avoids making dispatch async, which would complicate the reducer.

**Decompose ↔ Reposition shared logic:** Both decomposition (one tile becomes two) and invalid-drop reposition (tile moves to a nearby spot) need to find a non-overlapping position near a target point. The `animateReposition` helper from 1.4.4 should be designed with this reuse in mind. Decomposition calls it twice (once for each new tile, offset from the original position).

**Hwatu animation feel:** Card games involve decisive physical motions — slapping cards down, flipping them over, sliding them across the table. Animations should be quick and confident, not floaty or bouncy. Suggested easings:

- `power2.out` for placements and slides
- `back.out(1.2)` for satisfying settles
- `power3.inOut` for flips and reveals
- Keep durations short: 150-300ms for most actions, 200ms stagger for reveals

**Particles:** Think subtle — a small spray of colored dots on compose, a gentle shimmer on correct reveal. Not confetti explosions. The 단청 aesthetic is refined, not exuberant.
